How the internet works

History of internet: US department of defense created advanced research project agency (ARPA) to keep its technology more advanced than the Soviets, to create a network that could survive a nuclear attack. Joseph Licklider convinced ARPA to fund research into a computer network to connect scientists and engineers throughout the country. 1969 Arpanet formed as a messaging service between UCLA, UCSB, Stanford, and University of Utah.

One of the first innovations by ARPAnet was packet switching as opposed to circuit switching .Packet switching allowed different computers to send messages along the same set of wires instead of each getting one. Arpanet used packet switching form the start sending them over phone lines. Stanford became the official record-keeper of everyone's addresses in 1973. Over 100 computers were connected by 1977. Hawaii was connected, then jumped over to England and Norway.

Other networks were being formed but the networks weren't able to talk to each other easily.

Transmission control protocol/Internet protocol (TCP/IP) was a standard way of formatting packets, so that everyone was speaking the same language. As all the different networks began to be connected to each other, that formed the Internet. Email was invented in 1971 and by 1973 made up more than 3/4 of all Arpanet's packets. The logistics of connecting everything was too difficult, so they came up with domain name system (DNS). Instead of each node having their own address, they divided things into a name system. At the top level was domain: instead of emailing john@example, you're emailing john@example.com. Within each domain, a host had a second domain (eg. mit.edu).

For a while, most users of the Internet were business, universities, governments.. official organizations rather than single users.

ARPANet's goal was government communication and the government wanted to move on, so in 1990, NSFNet officially replaced ARPAnet as the backbone of the internet to allow for more commercial goals. Congress passed a law in 1992 allowing commercial traffic on NSFnet 1989 we got the first Internet Service Providers (ISPs) that connected local people to the Internet In 1995, NSFnet shut down and handed everything over to ISPs to connect to a network your computer placed a phone call to the network through a modem, which translated between the digital signals used in computers and the analog signals used in landline phones However, phone lines weren't made to transmit high frequencies, which were what was needed to transmit large amounts of information. One of the things modems did was check how high of a frequency the phone line could transmit, and then slow down your request

Internet is a shortene form of the word "internetworking" coined in the 70s to refer to the physical cables and computers all connecting together

Tim Berners-Lee and Robert Cailliau wanted to improve arrangement of information on the internet away from a tree-like structure, and used hypertext, which was invented in the 60s to go from one part of a document to another He started using hypertext as a means to navigate what he started to call the World Wide Web with hyperlinks connecting different web pages. The web was a hit at CERN, and was made public in 1993 HTTP stands for Hypertext Transfer Protocol, set of programs to read documents with hypertext. // introduces what comes next. www means the page is part of the world wide web.

Netscape navigator released in 1994 and code incorporated eventually into firefox 90s had a lot of failed startups, and the dotcom bubble burst in 2000, when stocks in tech companies plummeted, over half of them declared bankruptcy, eventually losing trillions of dollars in total Napster had been one of the fastest-growing businesses in history, but went bankrupt paying back musicians for copyright infringement

Microsoft narrowly avoided being broken up after violating antitrust laws Geocities encouraged users to create their own websites, was third-most popular site on the web in 1999, behind AOL and Yahoo!, was bought by Yahoo By the time Myspace took over, people were using broadband instead of dial-up. Dial-up could only transmit 56 kilobits per second DSL, which transmits digital data along phone lines instead of analog signals, and cable, which uses wires for a cable box to connect to the internet, became known as broadband (basically everything that wasn't dialup), which were 10s to 1000s times faster than dialup, overtook dialup in 2005

Data centers continued to survive even after dotcom bubble burst, and data centers began to be used to run special program sto look through that data and predict what you want to see How the internet works in 5 minutes: the internet is a wire in the ground. servers are computers directly connected to this wire. web pages are fires on that server's hard drive. every server has a unique internet protocol address, or IP address. your computer at home is not a server, but is called a client, because it is connected indirectly to the internet through an ISP via something like DSL. everything connected directly or indirectly to the internet has an IP address, including computers, servers, cell phones, and any equipment in between. anywhere two or more parts of the internet intersect there is a router that directs packets around the internet. each time the packet you send passes through a router, it adds on the IP address of the router as a wrapper, and as information is sent back, that wrapper is taken off by each router to know where to send it to. Document object model (DOM)

The DOM is a programming interface for web documents. It represents the page so that programs can change the document structure, style, and content. The DOM represents the document as nodes and objects. It is an object-oriented representation of the web page.

The DOM is built using multiple APIs that work together. The core DOM defines the entities describing tany document and the objects within it. Examples of expansions are the HTML DOM API which supports representing HTML documents to the core DOM and the SVG API for supporting representing SVG documents.

The DOM was designed to be independent of any particular programming language, making the structural representation of the document available from a single, consistent API.

There are several fundamental data types. Document is the root object itself. Node is every object located within a document. Element refers to an element or a node of type element returned by a member of the DOM API. Instead of a method returning an object reference to a node, the method returns the element that has just been created by the DOM. A nodelIst is an array of elements. An Attr is an object reference that exposes a special interface for attributes. NamedNodeMap is like an array but the items are accessed by name or index.

Viewport

The viewport is the user's visible area of a web page. The meta viewport tag sets the width of the page to follow the screen-width of the device, and the initial scale sets the initial zoom level.

Rules include: Do NOT use large fixed width elements, because this will cause the viewport to scroll horizontally, which people are not accusotmed to. Do NOT let the content rely on a particular viewport width to render well. Use CSS media queries to apply different styling for small and large screens.

Mobile first means designing for mobile screens first, which suggests instead of changing styles when the width gets smaller, you should change styles when the width gets larger.

Typical breakpoints are at 600px (phones, portrait tablets), 768 px (landscape tablets), 992px (smaller laptops/desktops), and 1200px (larger laptops/desktops)

Introduction to server side

Static sites are sites that return the same hard-coded content from the server whenever a particular resource is requested. Dynamic sites generate response content dynamically, usually by inserting data from a database into placeholders in HTML templates, and can return different data depending on user or stored preferences and can perform other operations as part of returning a response such as sending notifications. Client-side code is primarily concerned with the user interface, and the server-side code is responsible for the business logic. The server-side code has full access to the server operating system. Web frameworks are collections of functions, object, rules, and other code constructs designed to solve common problems, speed up development, and simplify tasks. Server side programming can store information in a database, return data in a proper way for client-side web frameworks to process more easily, can provide customized user experience, can control access to content, store session/state information, allow for notifications and communication such as through email, SMS, etc, perform data analysis. To communicate with a web server, the web browser sends a HTTP request, which includes a header that contains information about the request and a body (often empty). The header contains information about the request, such as the type of request, the resource being requested, and the version of the protocol. The body contains the data that is being requested. The response contains a header that includes a response code, type of content, character set, how big the content is, etc. Server-side web frameworks can be selected with these factors: effort to learn aka familiarity with the underlying programming language (easiest is Django), productivity, performance, caching support, scalability, web security. Djagno was created to develop a newspaper website, so ti's good for blogs and publications. Flask, for example, is lighter-weight and great for web apps running on embedded devices. Opinionated frameworks are better at solving common problems but may have trouble being flexible to solve less common problems. Smaller frameworks can be run on more constrained environments but may require more effort to learn. Speed usually is not the most important factor for the server.

Website security

Website security is the act/practice of protecting websites from unauthorized access, use, modification, destruction, or disruption. If you're using a server-wide web framework, it will almost certaintly enable "by default" robust and well-thought-out defense mechanisms against a number of the more common attacks. Other attacks can be mitigated through web server configuration (eg. HTTPS), and there are vulnerability scanner tools to check for obvious mistakes.

Threats

Cross-site scripting (XSS) is a class of attacks that inject client-side scripts into the browsers of other users. A reflected XSS vulnerability occurs when user content passed to the server is returned immediately and unmodified for display in the browser. For example, being able to add scripts to URL parameters. A persistent XSS vulnerability occurs when the malicious script is stored on the website, eg. submitting a comment with malicious script through unmodified HTML. This is very popular as the attacker does not have any direct engagement with the victims. The best defense against XSS is to remove or disable markup that can run code, such as the script, object, embed, and link tags. This process of modifying user data to prevent scripts is known as input sanitization. Many web frameworks automatically sanitize user input from HTML forms by default. SQL injection attacks databases. The attacker inputs user input to a SQL statement that changes the meaning of that statement. For example, instead of entering their username, they can enter something that destroys the users table. To avoid this attack, you must ensure that any user data passed to an SQL query cannot change the nature of the query. One way to do this is t oescape all characters in the user input that hvae a special meaning in SQL. Cross-site request forgery (CSRF) is a type of attack where the attacker steals credential information of a user and uses that information to submit a request to a server. One way to prevent this type of attack is to require POST requests have a user-specific site-generated secret, supplied by the server, thereby preventing John from creating his own form. General strategies for security include input sanitization, effective password management (strong passwords, two-factor authentication), HTTPS (encrypts data), keep track of most popular threats, use vulnerability scanning tools, and only store and display data that you need (eg. display only 4 digits of credit card number).

The Twelve-factor App

Link to resource The twelve-factor app is a short overview on best practices for an app. 1. Codebase: Always track in a version control system such as Git, Mercurial, or Subversion. A copy of the revision tracking database is known as the code repository, shortened to code repo or just repo. A codebase is a single repo or any set of repos who share a root commit. A deploy is a running instance of the app. One codebase, one app with multiple deploys. 2. Dependencies: Always declare dependencies via a dependency declaration manifest and avoid implicit dependencies from leaking in from the surrounding system using a dependency isolation tool. For Python, this is done with Pip for declaration and Virtualenv for isolation. 3. Config: An app's config is everything that is likely to vary between deploys. Store config in environmental variables (or env vars or env). Avoid grouping into development, test, and production, and instead keep every environmental variable independent from each other to allow for scalability. 4. Backing services: A backing service is any service the app consumes over the network as part of its normal operation. Avoid making any distinction between local and third party services, so that you can swap out services easily. Each distinct backing service is a resource. Treat resources as attached resources, indicating a loose coupling to the deploy they are attached to. 5. Build, release, run: The build stage transforms a code repo into an executable bundle known as a build, which fetches vendors dependencies and compiles binaries and assets. The release stage combines the build with the deploy's current config, and produces a release that is ready for immediate execution in the execution environment. The run stage (or runtime) runs the app in the execution environment, by laucnhing some set of the app's processes against a selected release. Strictly separate between these stages. Every release should have a unique release ID. 6. Processes: The app is executed in the execution environment as one or more processes. Keep processes stateless and share-nothing. Put data that needs to persist in a stateful backing service, typically a database. Reserve the memory space of the process to brief, single-transaction caches such as downloading a large file, operating on it, and storing the results in a database. Never assume that anything cached in memory or on disk will be available for a future request or job. Avoid things like django-assetpackager which uses the filesystem as a cache for compiled assets, or "sticky sessions" which cache user session data in memory of the app's processes. Instead, cache assets during the build stage and use datastores with time-expiration like Memcached or Redis. 7. Port binding: Make your app self-contained and do not rely on runtime injection of a webserver into the execution environment to create a web-facing service. The web app exports HTTP as a service by binding to a port, and listening to requests coming in on that port. 8. Concurrency: Make processes a priority. Take cues form the unix process model for running service daemons, assigning each type of work to a process type to handle diverse workloads. Use a share-nothing, horizontally partitionable nature. The array of process types and number of processes of each type is known as the process formation. Never daemonize or write PID files. Instead, rely on the operating system's process manager to manage output streams, respond to crashed processes, and handle user-initiated restarts and shutdowns. 9. Disposability: Make processes disposable, such that they can be started or stopped at a moment's notice (when they receive a SIGTERM signal from the process manager). Develop robustness against sudden death. 10. Dev/prod parity: Keep development, staging, and production as similar as possible. This reduces time gap, personnel gap, and tools gap. Resist the urge to use different backing services between development and production (eg. SQLite in development and MySQL in production). 11. Logs: Logs provide visibility into the behavior of a running app and are the stream of aggregated, time-ordered events collected from output streams of all running processes and backing services. Never concern yourself with routing or storage of the output stream. Do not manage logfiles. Instead, write event streams, unbuffered, to stdout. 12. Admin process: Process formation is the array of processes used to do the app's regular business as it runs. Use languages that provide a REPL shell out of the box. Run one-off admin processes in an identical environment as the regular long-running processes of the app, against a release using the same codebase and config as any process against that release, and ship with application code to avoid synchronization issues.
